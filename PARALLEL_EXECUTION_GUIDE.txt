PARALLEL EXECUTION OPTIONS:
===========================

1. parallel="methods" (RECOMMENDED)
   - Runs each @Test method in a separate thread
   - Best for independent test methods
   - Each thread gets its own WebDriver instance

2. parallel="tests"
   - Runs each <test> block in parallel
   - Good for multiple test suites

3. parallel="classes"
   - Runs each test class in parallel
   - Useful when you have multiple test classes

4. parallel="instances"
   - Runs each test instance in parallel
   - For parameterized tests


THREAD COUNT CONFIGURATION:
===========================

Current: thread-count="4"

Adjust based on your system:
- Low-end: thread-count="2"
- Medium: thread-count="4" (Current)
- High-end: thread-count="8" or higher

More threads = More browser instances = Higher resource usage

WHY BASETEST IS PERFECT FOR PARALLEL EXECUTION:
================================================

@BeforeClass (per test method):
- Creates NEW WebDriver for each test
- Creates NEW ConfigReader for each test
- Runs BEFORE each @Test method

@AfterClass (per test method):
- Closes WebDriver for each test
- Runs AFTER each @Test method

IMPORTANT NOTES:
================

⚠️  Thread Safety:
- Each test has its own WebDriver instance
- ConfigReader is thread-safe for reading
- No shared state between tests

✓ Resource Requirements:
- Each parallel thread opens a browser
- 4 threads = 4 browsers running simultaneously
- Ensure system has enough RAM

✓ Logging:
- Each thread has its own System.out stream
- Output will be interleaved but labeled per thread
